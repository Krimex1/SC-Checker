Overview

SC Checker Go is a comprehensive web security scanner written in Go. It performs automated reconnaissance, vulnerability assessment, and security analysis of web applications and servers.

Tech Stack

LanguageGo 1.26
GUIFyne v2 (desktop)
HTTPnet/http with connection pool (24 concurrent clients)
ConcurrencyGoroutines
EncryptionAES-256-GCM for secrets
SSRF ProtectionDNS-pinning for webhooks

Key Capabilities

  • 35+ security checks — headers, SSL/TLS, cookies, CORS, HSTS, WAF detection
  • Injection testing — SQL, XSS, SSTI, CRLF, Host Header, Dir Traversal, Open Redirect
  • Port scanning — 34 common ports + custom, banner grabbing
  • Path brute-force — 8 framework wordlists (WordPress, Laravel, Drupal, Joomla, Spring, Django, Next.js, Node.js)
  • Advanced checks — JWT, GraphQL, WebSocket, HTTP Smuggling, Supply Chain, Email Security
  • Reconnaissance — DNS records, subdomains, WHOIS, Shodan, Certificate Transparency, IP geolocation
  • DSL engine — Custom rule language with AST-based parser and safety validation
  • Plugin system — JSON + external executable plugins, 9 lifecycle hooks
  • Webhook notifications — Discord, Slack, Telegram, Pushover, Custom (with SSRF protection)
  • Auto-update — GitHub-based with SHA256 verification

Features

Security Checks

HSTS, Clickjacking (X-Frame-Options + CSP frame-ancestors), CORS (wildcard + origin reflection + credentials), Cookie audit (HttpOnly/Secure/SameSite for sensitive cookies), Mixed Content detection (with 40+ tracking domain exclusions), HTTP→HTTPS redirect, TRACE method, CSP analysis, Permissions-Policy, Referrer-Policy, Directory listing, security.txt discovery.

SSL/TLS Analysis

Certificate expiry (days remaining), weak cipher detection (RC4/DES/3DES/NULL/EXPORT/MD5), weak protocol detection (≤TLS 1.1), Subject/Issuer/SAN extraction, serial number, deep SSL info.

Port Scanning

34 common ports: 21(FTP), 22(SSH), 25(SMTP), 53(DNS), 80(HTTP), 443(HTTPS), 3306(MySQL), 5432(PostgreSQL), 6379(Redis), 27017(MongoDB), 3389(RDP), etc. Banner grabbing with protocol-specific probes (HTTP, SMTP). Custom ports from plugin lists.

WAF Detection (14 signatures)

Cloudflare, AWS WAF, ModSecurity, Imperva/Incapsula, Akamai, Sucuri, Wordfence, Barracuda, F5 BIG-IP, FortiWeb, DenyAll, Radware, Citrix, Kona. Both header and body-based detection.

CMS/Framework Detection (19 signatures)

WordPress, Joomla, Drupal, Laravel, Django, Spring, Express, Next.js, Nuxt.js, Angular, ASP.NET, Ghost, Shopify, Magento, Ruby on Rails, Flask, Symfony, React, Vue.js.

Version Fingerprinting (37 patterns)

Nginx, Apache, PHP, IIS, LiteSpeed, OpenResty, Caddy, Varnish, Tomcat, Jetty, Undertow, MySQL, MariaDB, PostgreSQL, WordPress, Drupal, Joomla, jQuery, React, Vue.js, Angular, Bootstrap, Node.js, Express, Python, Ruby, Rails, cURL, Wget.

Reconnaissance

DNS records (MX, NS, TXT), subdomain enumeration (80+ wordlist), reverse DNS, zone transfer check, email extraction, phone number extraction, social link detection, external link collection, JS library detection (jQuery, React, Vue, Bootstrap, Lodash, Axios), meta tag extraction, hidden form discovery.

External Services

IP geolocation (ip-api.com), ASN lookup (ipinfo.io), WHOIS (rdap.org + TCP fallback), Shodan (internetdb.shodan.io), Certificate Transparency logs (crt.sh).

Scan Workflow

The scanner executes 28 stages in a deterministic pipeline:

 1. Target normalization        (validate URL, block private IPs)
 2. DNS resolution              (A/AAAA record lookup)
 3. Connectivity probe          (TCP dial on primary + alt ports)
 4. Server node detection       (non-HTTP hosts get TCP-only scan)
 5. Root HTTP probe             (initial request, status, redirect)
 6. Alternative port probing    (8080, 8443, 8000, 5000, etc.)
 7. Header collection           (all response headers → map)
 8. Security checks             (headers, cookies, WAF, CMS, versions)
 9. Concurrent checks           (methods, CORS, HTTPS redirect, XSS, SQL)
10. SSL/TLS check               (certificate, cipher, protocol)
11. DNS & recon checks          (MX, NS, TXT, subs, reverse, robots, sitemap)
12. Port scanning               (34+ ports, concurrent TCP dialers)
13. Path brute-force            (500+ paths across 8 frameworks)
14. Extraction                  (emails, phones, social, JS libraries)
15. Injection checks            (CRLF, Host Header, Open Redirect, Dir Traversal)
16. Leak detection              (backups, source leaks, admin panels, login pages)
17. Mutation tests              (payload variants against SQL/XSS endpoints)
18. HTTP methods full           (GET, HEAD, POST, PUT, DELETE, PATCH, OPTIONS, TRACE)
19. Exploit verification        (re-test with stronger payloads)
20. Performance metrics         (TTFB, content size, encoding, redirect chain)
21. CSP/security.txt analysis
22. Advanced checks             (JWT, GraphQL, WebSocket, Supply Chain, etc.)
23. DSL evaluation              (v2 AST + v1 rules)
24. CVSS scoring                (severity ratings for findings)
25. Recon services              (ip-api, ipinfo, rdap, Shodan, crt.sh)
26. Anomaly hints               (summary of noteworthy findings)
27. Risk scoring                (0–100 score with level classification)

CORS Testing

Tests wildcard origin (*), origin reflection, and credentials exposure against https://evil.com and null origins.

Cookie Audit

Only flags sensitive cookies (session, auth, CSRF, JWT tokens — 20+ patterns). Checks HttpOnly, Secure, and SameSite attributes. Tracking/analytics cookies are ignored.

Mixed Content

Finds http:// resources in <script>, <img>, <iframe>, <link>, <video>, <audio>, <source>, <embed>, and <object> tags. Excludes 40+ known tracking/CDN domains to reduce false positives.

SSL/TLS

  • Certificate expiry: <30 days = HIGH
  • Weak ciphers: RC4, DES, 3DES, NULL, EXPORT, MD5
  • Weak protocols: TLS 1.0, TLS 1.1
  • Subject, Issuer, SAN, serial number extraction

Injection Testing

SQL Injection

Injects ' and %27 into query parameters. Detects 7 SQL error indicators (MySQL, PostgreSQL, MSSQL, Oracle, SQLite). Score: +10 per finding.

XSS Reflection

Injects a unique token, strips <script>, <style>, <noscript>, attributes, URLs, meta tags, and comments from the response before checking for reflection. Score: +10.

SSTI (Server-Side Template Injection)

Tests 4 template engines with payloads expecting mathematical evaluation:

Jinja2/Twig{{7*7}}Expects 49
FreeMarker/Velocity${7*7}Expects 49
ERB<%= 7*7 %>Expects 49
Ruby/Slim#{7*7}Expects 49

CRLF Injection

Injects %0d%0a into URL paths and query parameters. Score: +6 per finding.

Host Header Injection

Spoofs Host header to detect server-side request forgery or header injection vulnerabilities. Score: +5.

Open Redirect

Tests redirect, url, next, return parameters with external URLs. Score: +5.

Directory Traversal

Tests ../../../etc/passwd and ..\..\..\windows\win.ini path traversal patterns.

Payload Mutation Engine

SQL and XSS payloads are mutated through 10 variant generators before testing:

  • URL encoding (single + double)
  • HTML entity encoding
  • Comment-space injection
  • Null byte prefix/suffix
  • Backtick substitution
  • Case swapping

Critical Path Validation

15 critical paths have content validation rules to eliminate false positives. For example:

  • .env — must contain =, APP_, DB_, SECRET, KEY=, or PASSWORD
  • id_rsa — must contain BEGIN, PRIVATE KEY, ssh-rsa, or -----BEGIN
  • actuator/env — must contain {, spring, server.port, or environment
  • .git/config — must contain [core], repositoryformatversion, or [remote

Soft-404 Filtering

10 soft-404 signatures prevent false path discoveries: "page not found", "404 not found", "no results found", etc. Paths matching these patterns are excluded unless critical content rules validate them.

Advanced Checks

JWT Analysis

Scans response headers and body for JWT tokens. Base64-decodes header and payload. Checks for:

  • alg: none — CRITICAL
  • alg: HS256 — brute-forceable
  • Sensitive data in payload (passwords, keys, tokens)

GraphQL Scanning

Probes /graphql, /api/graphql, /gql. Sends {__typename} to detect. Runs introspection query for full schema. Detects: introspection enabled, deprecated fields, sensitive mutations, debug mode.

WebSocket Testing

Probes /ws, /socket, /websocket (ws:// and wss://). Tests: XSS payload reflection, template injection (SSTI), 65KB DoS payload.

HTTP Smuggling

Tests CL.TE and TE.CL request smuggling: manipulates Content-Length and Transfer-Encoding headers to detect desync vulnerabilities. Score: +10 per finding.

Supply Chain Analysis

Extracts external <script> and <link> URLs. Checks: HTTP vs HTTPS, CDN identification, vulnerable versions (jQuery <1.7, AngularJS 1.x, Moment.js, Bootstrap 3, Lodash 4), CORS wildcard.

Email Security

Checks SPF, DMARC (warns on p=none), DKIM (6 selectors: default, google, selector1, k1, dkim, mail).

Subdomain Takeover

Checks subdomains against Heroku, GitHub Pages, Surge.sh, AWS S3 takeover signatures.

SSTI (Server-Side Template Injection)

Tests Jinja2/Twig, FreeMarker/Velocity, ERB, Ruby/Slim engines via GET and POST.

Other Checks

  • Hidden Endpoints — extracts URLs from JS strings and API patterns
  • Rate Limiting — 10 rapid requests to detect 429/503

DSL Rules Engine

DSL v1 — JSON Rule Format

Define conditions in JSON that are evaluated against the scan report:

[
  {
    "name": "No HSTS",
    "condition": "hsts_enabled == false",
    "severity": "HIGH",
    "message": "HSTS not enabled"
  },
  {
    "name": "SSL Expiring",
    "condition": "ssl_expiry_days < 30",
    "severity": "HIGH",
    "message": "SSL certificate expires within 30 days"
  }
]

DSL v2 — AST-Based Mini-Language

A full programming language parsed by a recursive-descent parser with AST safety validation:

IF critical_paths_count > 0 THEN
  ASSERT risk_score > 20
ELSE
  ASSERT risk_score >= 0
END

IF open_ports_count > 10 THEN
  ASSERT risk_score > 10
END

IF ssl_weak_cipher == true THEN
  ASSERT risk_score > 30
END

FOR $port IN open_ports
  ASSERT $port > 0
END

FOR $sub IN subdomains
  CAPTURE "www" FROM target
END

DSL v2 Language Reference

$var = exprVariable assignment
IF cond THEN ... ELSE ... ENDConditional block
FOR $item IN list ... ENDLoop over report list field
ASSERT condAssertion (fails = HIGH severity)
CAPTURE "regex" FROM fieldRegex capture from report field
REQUEST "url" CHECK RESPONSE CONTAINS "text"HTTP request with content check
HTTP_TIME "url" <|> msResponse time threshold check
RETURNEarly return from program

Available Fields

Boolean: hsts_enabled, http_to_https_redirect, clickjacking_protected, xss_reflection, trace_enabled, ssl_weak_cipher, mixed_content, server_node

Numeric: risk_score, status_code, ssl_expiry_days, response_time_ms, scan_duration_ms, port

String: target, ip, host, risk_level, server_banner, normalized_url

Counts: critical_paths_count, open_ports_count, subdomains_count, sql_errors_count, cors_issues_count, cookie_issues_count, cve_findings_count, ssti_results_count, graphql_vulns_count, admin_panels_count, source_leak_count, backup_files_count, etc.

AST Safety Validation

Before execution, the AST is validated:

  • All field references checked against a whitelist of 30+ allowed fields
  • REQUEST URLs blocked if they contain localhost, 127., or @
  • CAPTURE regex patterns validated at parse time
  • Unknown node types rejected
  • Unknown operators rejected

User-Agent Rotation

When useragents.txt contains entries, the HTTP client rotates through them — each request uses the next UA. Every 8 requests, a random jump is made (to avoid pattern detection). If the list is empty, the default UA is used: Mozilla/5.0 (compatible; SC-Checker-Go/1.0).

SSRF Protection (DNS-Pinning)

All webhook requests use DNS-pinning to prevent Server-Side Request Forgery:

  1. Resolve webhook hostname to IP via DNS
  2. Validate all resolved IPs — reject loopback, private, link-local
  3. Connect directly to the pinned IP (not hostname)
  4. Set original Host header on the HTTP request
  5. Custom TLS config with InsecureSkipVerify for pinned connections

Secret Encryption

All webhook URLs are encrypted using AES-256-GCM before storage. The prefix enc: in webhooks.json indicates encrypted values. Legacy plaintext values are read transparently (backward compatible).

Auto-Update

Update Check

On startup, SC Checker checks the latest GitHub release. If a newer version is available, a dialog shows the changelog and offers to update.

SHA256 Verification

Before installing, the downloaded binary is verified against the SHA256 checksum from the release assets. If the hash doesn't match, the update is refused — preventing corrupted or tampered downloads.

Update Process (Windows)

  1. Download new .exe to a temp file
  2. Verify SHA256 checksum (required)
  3. Generate a batch script that:
    • Waits 2 seconds (for the app to exit)
    • Renames old .exe → .old
    • Moves new .exe into place
    • Launches the updated app
    • Self-deletes the batch script
  4. On failure: restores .old and launches original version

Requirements

  • GitHub repository: Krimex1/SC-Checker
  • Releases must include the .exe asset
  • Optional: sha256sums.txt for hash verification
  • Version format: vX.Y.Z (semver)
